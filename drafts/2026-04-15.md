### Post
Always-on agents are the easy part. The hard part is getting them to fail without taking your own API quota down with them.

Anthropic shipped Routines in Claude Code on Monday. Cloud-hosted agents triggered by schedule, API, or GitHub event. They run while you sleep. The demo ships beautifully.

Meanwhile AgentBrief reported a 40% failure rate on production multi-agent systems this week. The mechanism was retry storms: one agent fails a tool call, retries, cascades the retry into every agent downstream, and suddenly your orchestration layer is DDoS-ing your own API. Shopify just turned on MCP write access to 5.6M stores. Vercel says 30% of deploys are now agentic. The blast radius of a retry storm is no longer theoretical.

Builders keep framing this as a tradeoff. Autonomy on one axis, reliability on the other, pick where to sit. The production runtime wins on both, and the shape of how it wins is specific.

A production agent runtime needs three properties at once. It runs without supervision on the happy path. It fails closed, not retry forever, when a tool returns an error it cannot reason about. And it escalates to a human at the exact boundary where confidence drops below a threshold, with enough context that the operator can decide in seconds, not minutes.

Anthropic Managed Agents, shipped last Thursday, moves in this direction: sandboxing, permissions, state, error recovery handled by the platform. Routines is the trigger layer on top. Cursor's parallel agent tiles this week push the same envelope on the IDE side. The missing piece is the gateway between the agent and the tool, where retry policy, rate limits, and graceful handoff back to the operator get enforced.

Always-on is a feature. Failing safely to a human is the moat.
